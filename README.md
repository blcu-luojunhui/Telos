# better me

**How to be a better me.**

为「我自己」打造一个长期可用的个人成长与生活操作系统：覆盖**成为更好的自己**所涉及的多个维度，在一个系统里做记录、记忆、规划和即时建议。

---

## 项目目标

### 愿景：多维度「更好的我」

系统面向的是**全人**，而不是单一领域。涵盖的维度包括（并会随实践拓展）：

| 维度 | 说明 | 示例目标/记录 |
|------|------|----------------|
| **吃** | 饮食与营养 | 三餐记录、热量/营养素、饱腹感、与状态的关系 |
| **锻炼** | 运动与身体 | 跑步/篮球/力量等训练、负荷、恢复、身体指标 |
| **读书** | 阅读与输入 | 在读/读完的书、笔记、摘抄、与思考/输出的连接 |
| **思考** | 反思与心智 | 日记、复盘、决策记录、困惑与顿悟、心智模型 |
| **工作** | 效率与产出 | 要事、专注时间、项目进度、精力与工作匹配 |
| **社交** | 关系与连接 | 重要的人、见面/联系频率、社交能量与边界 |
| **睡眠** | 休息与恢复 | 睡眠时长与质量、作息规律、与情绪/表现的关系 |
| **情绪** | 心理与状态 | 心情、压力、动力、焦虑/平静，与各维度的交叉 |
| **学习** | 技能与成长 | 在学技能、练习记录、里程碑、与读书/工作联动 |
| **兴趣/创造** | 玩与创造 | 爱好、 side project、放松方式、心流体验 |

一套**记忆、规划、目标、主观体验**的哲学

### 核心能力（以「吃 + 锻炼」为起点）

- **长期记忆**：记住我的训练记录（长跑、打篮球等）、饮食记录和身体指标变化；未来扩展至阅读轨迹、工作节奏、社交模式等。
- **周期性规划**：按月/按周生成训练+饮食计划，并根据执行情况自动调整；后续可做阅读计划、复盘节奏、工作 OKR 等。
- **即时建议**：根据当天的运动 / 饮食 / 状态，给出「今天/今晚/明天」的具体建议；可延伸到「今天适合深度工作还是沟通」「今晚要不要社交」等。
- **目标驱动**：支持设定目标（如「16 周后完成半马」），围绕目标生成周期化训练+饮食方案；同一套目标系统可承载读书目标、项目 deadline、习惯目标等。

## 设计哲学：系统在学什么、优化什么

### 系统学习的对象是「我这个人」，不是模板

普通 App 的范式是：背后有一堆训练/饮食模板，按你填的问卷参数从库里挑一个最合适的给你。这里的定位不同：**学习对象从「某个计划模板」升级成「你这一个人」**——包括你的训练、饮食、读书、思考、工作、社交、睡眠、情绪等如何相互影响。

- **输入**：训练负荷、饮食结构、生活压力/睡眠、时间（后续加入阅读量、工作负荷、社交频率等）。
- **输出**：你的主观状态（疲劳、心情、动力）+ 客观指标（体重、体脂、配速、心率等）。

也就是说，它要学的是：给你某种「周跑量 + 配速结构 + 饮食模式」组合后，一周、两周、四周后，你的身体和感受会怎样——好还是差。
靠经验规则，后者用长期数据 + 记忆 + 简单统计/LLM，做**个体级别的自适应系统**。

### 优化的是一个多目标函数，而不是单一 KPI

目标不只是「瘦 10 斤 / 跑半马」这样的终点，而是显式定义一个**体验与结果兼顾的目标函数**，例如：

- **G₁**：体重/体脂朝目标区间移动。
- **G₂**：跑步表现（配速、完赛时间）。
- **G₃**：长期可持续性（不崩盘、不讨厌训练、不报复性反弹）。
- **G₄**：日常幸福感（心情、压力、睡眠感受）。

系统做的不是「最大化某一个」，而是在你设定的权重下做权衡，例如：

- 半马备赛期：提高 G₂ 的权重；
- 比赛结束后：提高 G₃、G₄ 的权重。

未来甚至可以由 agent 根据你的历史行为，反推你「嘴上说想减脂、实际更在意幸福感」的真实权重，并做温和引导。

### 终极形态：个人级「更好的我」OS，而不是又一个 App

- **Keep** 是平台产品：帮很多人用统一模板变好一点。
- 这个项目的终极形态是：**你私有的「better me」操作系统**——覆盖吃、锻炼、读书、思考、工作、社交、睡眠、情绪等多维度的**私人教练 + 生活顾问**，为「你的一生」积累经验知识。跨年跨月的长程记忆，用来回答：哪几年你状态最好、那时候的训练–饮食–工作–社交–心情组合是怎样的；哪些阶段你反复崩盘或停滞、背后的模式是什么。当前从「吃 + 锻炼」落地，架构上为其它维度留好扩展位。

## 总体架构概览

整体分为 4 层 + 1 条时间轴：

1. **数据层（事实记录）**
2. **知识与记忆层（规则 + 长期记忆）**
3. **决策与规划层（agent 大脑）**
4. **交互层（我和 agent 怎么对话）**
5. **时间维度（每天/每周/每阶段循环）**

### 1. 数据层（事实记录）

使用 MySQL 存储结构化数据，Redis 做缓存，Milvus/ES 做长期记忆检索。

- **训练记录 `workouts`**
  - 字段示例：
    - `id`, `date`, `type`（run / basketball / strength / other）
    - `duration_min`, `distance_km`, `avg_pace`, `avg_hr`, `calories`
    - 主观体验（一等公民）：`subjective_fatigue`（1–10）, `sleep_quality`（1–10）, `mood`, `motivation`, `stress_level`
    - `note`

- **身体指标 `body_metrics`**
  - 字段示例：
    - `id`, `date`
    - `weight`, `body_fat`, `muscle_mass`
    - `resting_hr`, `bp_systolic`, `bp_diastolic`
    - `sleep_hours`

- **饮食记录 `meals`**
  - 起步阶段可以简化为：
    - `id`, `date`, `meal_type`（breakfast / lunch / dinner / snack）
    - `food_items`（自由文本）
    - `estimated_calories`, `protein_g`, `carb_g`, `fat_g`（可选）
    - 主观体验：`satiety`（1–10）, `mood`（1–10）, `stress_level`（可选）

- **个人配置与偏好 `user_profile`**
  - 静态/低频数据：
    - `height`, `gender`, `birth_year`
    - `activity_level`（sedentary / moderate / high）
    - 饮食偏好与忌口（是否吃辣、牛奶、牛肉、猪肉等）

- **目标系统 `goals`**
  - 所有目标（减脂、增肌、维持、半马等）统一用一个 Goal 模型表示，只是类型和 `target` 结构不同。
  - 通用字段：
    - `id`
    - `type`：`weight_loss` / `muscle_gain` / `maintenance` / `race` / ...
    - `target`：按类型不同存结构化目标内容（见下）
    - `deadline`：目标截止日期（例如今天 + 3 个月 / 6 个月）
    - `status`：`planning` / `ongoing` / `completed` / `abandoned`
    - `created_at`, `updated_at`, `note`
  - 减脂目标示例（weight_loss）：
    - `type = weight_loss`
    - `target = { start_weight, target_weight, expected_loss }`
    - 例如：「6 个月瘦 10 斤」→ `start_weight = 72kg, target_weight = 67kg, expected_loss = 5kg`
  - 比赛型目标示例（race）：
    - `type = race`
    - `target = { race_type, target_time, weekly_time_budget, constraints }`
    - `race_type`（`half_marathon` / `10k` / `full_marathon` 等）
    - `race_date`
    - `target_time`（如 1:45:00，可选）
    - `weekly_time_budget`（每周可训练小时数）
    - `constraints`（伤病、工作限制、场地限制等）
  - 支持同时存在多个 active goals（例如「3 个月挑战半马」+「6 个月瘦 10 斤」），由上层规划模块综合考虑。
  - **目标轨迹（Trajectory）**：例如减脂目标不只是「6 个月瘦 5kg」一个点，而是规划一条合理体重变化曲线（前期稍快、中期稳定、后期保守）；每周复盘时对比「实际曲线 vs 目标曲线」，偏差大时再判断是饮食、活动量还是压力睡眠问题，做针对性调整。
  - **目标层级与策略切换**：当多个目标冲突时（如备赛后期 vs 减脂），系统有显式规则——例如比赛前 4 周性能优先、比赛后 2–4 周恢复与体重并重——实现策略切换，而不是一条线撸到底。

### 主观体验最主要的

**把主观体验当作一等特征**：真正想要的是长期可持续的高质量生活方式，可持续性的核心变量是——你觉得这件事痛不痛苦、累不累、烦不烦、有没有成就感。

- **数据层**：除客观指标外，刻意记录：
  - `subjective_fatigue`（训练后/次日）
  - `mood`（整体心情）
  - `motivation`（对训练/饮食的意愿）
  - `stress_level`（工作/生活压力）
- **记忆与规划**：Memory Writer 总结时，不只写「这周跑了 40km、体重降 0.3kg」，也写「这周训练+饮食较严，主观上非常疲惫、社交受影响，下次不要把这种组合连续超过 2 周」。规划时，对历史上让你主观体验很差的训练+饮食模式，自动降低使用频率或限制连续使用时间。

### 2. 知识与记忆层

分为「通用知识」和「个体化记忆」两部分。

- **通用知识（规则/公式）**
  - BMR / TDEE 估算公式。
  - 减脂 / 维持 / 增肌 的热量区间策略。
  - 蛋白质摄入建议（g/kg 体重）。
  - 不同训练日类型（休息日 / 轻松日 / 关键课）对应的饮食策略。
  - 实现方式：
    - 一部分写成代码公式（确定和可重复）。
    - 一部分写入 LLM 的系统提示词（营养与训练原则）。

- **个体化长期记忆**
  - 用 Milvus/ES 存储「记忆卡片」，如：
    - 某一阶段的训练/饮食/体重变化总结。
    - 某些模式：如「高跑量 + 低碳水时主观状态很差」。
  - 每日/每周有一个 Memory Writer：
    - 从训练/饮食/对话中抽取摘要，生成向量嵌入，写入记忆库。
  - agent 做规划时：
    - 先基于当前话题（例如「半马备赛」「减脂」）从记忆库检索相关记忆。
    - 把这些记忆连同近期数据一起输入 LLM。
  - **失败经验的系统总结**：不只记录成功周期，也把失败周期（没完成备赛、减脂中途崩掉）总结成结构化记忆，例如「当时失败主要是因为工作强度突增 + 饮食过于激进 → 持续 2 周后崩盘」。下次遇到类似背景（如工作又开始爆炸）时，agent 会提醒你上次在类似环境下极端计划是怎么崩的，并主动给出更温和的方案。

### 3. 决策与规划层（agent 大脑）

核心有三个层级：

1. **长期与阶段性规划（目标驱动）**
   - 输入：
     - 当前 active goals（如「16 周后跑半马」+「3 个月减脂」）。
     - 最近 3–6 个月训练/身体指标数据。
   - 输出：
     - 阶段划分（基础期 / 提升期 / 峰值期 / 减量期）。
     - 每个阶段的周跑量区间、关键训练课类型比例。
     - 每个阶段的体重/体脂/宏观营养方向。

2. **周计划生成器 `WeeklyPlanner`**
   - 输入：
     - 短期目标（例如「本周是第 5/16 周，处于提升期」）。
     - 最近 2–4 周的训练数据和疲劳状态。
     - 本周日程（哪些天可训练、是否有固定打球时间等）。
   - 输出（结构化 JSON）：
     - 每天的：
       - `day_type`：`rest_day` / `easy_day` / `key_workout_day`。
       - `workout_plan`：训练类型、距离/时间、配速/强度范围等。
       - `nutrition_target`：目标热量、蛋白质/碳水/脂肪区间。
   - 周计划同时输入「训练模块」和「饮食模块」。

3. **日内即时调整 `DailyAdjuster`**
   - 输入：
     - 当天已经完成的训练和饮食记录。
     - 当前主观状态（疲劳、睡眠质量、情绪）。
   - 输出：
     - 当天剩余时间的战术级决策：
       - 晚饭吃什么方向，是否加餐。
       - 是否降低/提高明天训练强度。
     - 必要时对本周后续计划做微调（如超负荷则减量）。

> 训练与饮食强绑定：  
> `day_type` 决定当天饮食策略：  
> - 关键课日：高碳水、适中热量，优先保证训练质量和恢复。  
> - 休息日：略低热量、略低碳水，推进体重/体脂目标。

### 4. 交互层

为单一用户（我自己）设计，从简单到复杂：

1. **CLI / Notebook 交互（MVP）**
   - 通过命令行/Notebook 与 agent 对话：
     - 例如：「帮我看下这周计划」「今天晚上打球，晚饭吃什么」。

2. **个人 Web 控制台（后续）**
   - 页面包含：
     - 体重/体脂/周跑量等趋势图。
     - 当前阶段目标与周计划卡片。
     - 一个对话窗口与 agent 互动。

3. **提醒与推送（再后续）**
   - 通过微信/Telegram/飞书机器人等：
     - 每天早晨推送「今日计划」。
     - 每天晚上推送「今日总结 + 明天建议」。

## 半马目标示例（结构设计）

以「16 周后挑战半马」为例：

- **创建目标（自然语言 → 结构化）**
  - 对话示例：
    - 「我想在 2025 年 11 月跑一个半马，目标 1 小时 45 分钟，我现在每周大概能跑 3 次。」
  - 解析为 `goals` 表的一条记录：
    - `type = race`
    - `race_type = half_marathon`
    - `race_date = 2025-11-xx`
    - `target_time = 1:45:00`
    - `weekly_time_budget ≈ 3–4 次训练`

- **宏观周期（16 周备赛）**
  - 阶段划分：
    - 第 1–4 周：基础有氧期（建立跑量和习惯）。
    - 第 5–10 周：提升期（加入更多节奏跑和间歇）。
    - 第 11–14 周：峰值期（接近比赛配速训练）。
    - 第 15–16 周：减量期（减少跑量，保持状态）。

- **周计划结构（示例 JSON 抽象）**
  - 每周：
    - `week_index`（第几周）
    - `phase`（基础 / 提升 / 峰值 / 减量）
    - `weekly_mileage_target`（km 区间）
    - `days[]`：
      - `date`
      - `day_type`（rest_day / easy_day / key_workout_day）
      - `workout_plan`（如「轻松跑 8km @ 6:00–6:15/km」）
      - `nutrition_target`（热量 + 三大营养素范围）

这些结构会由 `RacePlanner` / `WeeklyPlanner` 等模块生成，并写入数据库，供 CLI/Web/提醒系统使用。

## 时间维度与运转方式

- **每日循环**
  - 记录当天训练、饮食、身体状态。
  - `DailyAdjuster` 根据偏差给出当天/明天的调整建议。
  - `Memory Writer` 抽取当天关键事件写入长期记忆。

- **每周循环**
  - 对本周执行情况复盘（训练达成度、体重变化、主观状态）。
  - 调整下一周的训练+饮食计划。

- **每阶段/每个目标结束后**
  - 生成更高层次总结（如「这次 16 周半马备赛的经验教训」）。
  - 存入记忆库，以便下次设定类似目标时自动参考。

## 实现路线（从 MVP 到完整系统）

1. **MVP**
   - 使用 CLI/Notebook 与 agent 互动。

3. **Web 控制台与提醒**
   - 搭建简单前端展示趋势和计划。
   - 添加每日/每周推送渠道（微信/Telegram 等）。

## 和市面运动 App 的差异与创新点（草案）

相对于 Keep 等大众运动 App，这个项目更像是「只为我一个人的私人教练中台」，有一些可以持续迭代的创新方向：

- **1. 目标组合与权重而非单一计划**
  - 市面 App 通常一次只跟一个训练计划（例如「半马训练营」或「减脂计划」）。
  - 这里的设计是：允许同时存在多个目标（如「3 个月半马」「6 个月瘦 10 斤」），由 `WeeklyPlanner` 在生成计划时自动做权衡：
    - 关键训练日优先保证能量和恢复。
    - 在不影响训练质量的前提下，长期推进减脂等体型目标。

- **2. 深度个体化的长期记忆**
  - 常见 App 的「智能」主要停留在模板 plan 上，个性化有限。
  - 这里通过 Memory Writer + 向量记忆，把「我过去的反应模式」长期积累起来：
    - 哪种跑量/配速组合我容易累垮。
    - 哪类饮食搭配我更有精神、更容易坚持。
  - 下次规划类似阶段时，会优先参考「自己的历史经验」，而不是公共模板。

- **3. 计划与现实的闭环调参**
  - 不只是「给一个 16 周计划」，而是每天/每周根据实际执行情况（训练完成度、体重变化、主观疲劳）动态调整：
    - 如果连续几周体重不降，就适度调整热量缺口或提高活动量。
    - 如果近期压力/疲劳过高，就自动降低训练强度或安排减量周。

- **4. 训练 + 饮食真正一体化**
  - 多数 App 把「训练计划」和「饮食记录」作为两个相对独立的模块。
  - 这里从架构上让它们共享同一个 `day_type` 和 `goal` 系统：
    - 关键课日 → 高碳水、足够热量。
    - 休息日 → 略低碳水、控制总热量。
  - 可以在同一个视图里看到「训练负荷–饮食–体重/体脂」之间的因果关系。

- **5. 针对「一个人」不断进化的教练风格**
  - 可以在对话与记忆中逐渐学习我的沟通偏好：
    - 更喜欢硬核数据还是温柔鼓励。
    - 更容易坚持的是严格打卡还是宽松建议。
  - 未来可以考虑让 agent 的「语言风格」「给建议的频率和方式」都随时间微调，真正像一个长期陪跑的教练。

- **6. 失败经验与长程复盘**
  - 系统会显式总结「失败周期」并写入记忆：为什么某次备赛或减脂中途崩掉（工作、饮食过于激进、伤病等）。在类似情境再次出现时，优先检索这些记忆，避免重复踩坑。
  - 跨年跨月的长程记忆，用来回答：哪几年状态最好、当时的训练–饮食–生活组合是什么；哪些阶段反复受伤或暴饮暴食、背后模式是什么。本质是在为「一生」积累经验知识图谱，而不是单次训练周期。
