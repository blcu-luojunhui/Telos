## 1. 产品愿景

### 1.1 核心定位

**Telos 是一个只属于我的私人 Agent。**
它记录我的生命，理解我的状态，承接我对世界的观察与感悟，沉淀我的认知，并帮助我在复杂、孤独而短暂的人生中，慢慢建立属于自己的秩序。

**命名来源**: Telos (τέλος) 是希腊语，意为"目的"、"终极实现"、"内在完成"。在亚里士多德哲学中，它指事物的内在目的和最终形态。这个名字直接对应产品的核心命题：帮助我成为我想成为的人。

**Slogan**:
- 中文：**记录生命，沉淀认知，建立秩序**
- 英文：**Become who you are meant to be.**

### 1.2 产品使命

让"我"拥有一个真正懂自己的长期助手，不只是帮助我记录和管理生活，更帮助我理解自我、观察世界、形成认知，并实现持续的自我进化。

### 1.3 核心价值主张

- **记录生命**: 记录日常行为、状态变化、关键经历与长期轨迹
- **理解自我**: 从事实、情绪、行为与选择中逐渐理解"我是谁"
- **观察世界**: 承接我对外部世界的观察、感受、判断与感悟
- **沉淀认知**: 将碎片化经历和思考逐渐沉淀为更稳定的认知结构
- **建立秩序**: 帮助我在复杂与变化中形成自己的原则、方法和长期方向

## 2. 产品定位与使用场景

### 2.1 产品定位

**Telos 是一个 Self-Use 产品**，当前阶段的唯一用户是产品创造者本人。

这不是一个面向大众市场的 toC 产品，而是一个**深度个性化的私人工具**。产品的设计、开发、迭代都围绕一个核心问题：

> **"如何在自我、生活、世界、认知与秩序这五个层面，持续理解我、支持我，并帮助我成为更好的自己。"**

### 2.2 为什么是 Self-Use

**1. 因为"我"本身就是最复杂、最长期、也最值得被理解的对象**
- 我的生命不是一个标准化需求集合，而是持续变化的整体
- 很多真正重要的问题无法被通用产品抽象成标准功能
- 只有 self-use，系统才能围绕"我是谁"持续演化

**2. 因为生命数据需要极高信任**
- 这里记录的不只是任务和健康，还包括孤独、恐惧、意志、观察、感悟与认知
- 这些内容比普通效率数据更私密、更脆弱
- 自己的数据自己掌控，系统才能真正承接真实的生命材料

**3. 因为这是一个长期生长的系统**
- 它的价值不在单次使用，而在长期积累
- 数据越完整，Agent 越能理解我的节律、模式、问题与成长轨迹
- 它不是一次性交付的工具，而是长期共生的系统

**4. 因为我自己就是最直接的验证场**
- 我能第一时间判断这个建议有没有价值
- 我能感受到这个系统是否真的懂我
- 我能快速迭代，而不被通用性约束拖慢

### 2.3 核心使用场景

**场景 1: 日常记录**
```
我: "今天跑了5公里，用时30分钟"
系统: "记录！配速6分钟/km，比上次快了10秒 💪"
```

**场景 2: 主动提醒**
```
系统: [早上8点]
     "早上好！昨天睡了6小时，比平时少1.5小时。
      今天建议轻量训练或休息，别硬撑。"
```

**场景 3: 跨维度洞察**
```
我: "最近感觉效率很低"
系统: "看了一下你最近的数据:
      - 连续5天睡眠不足7小时
      - 运动频率从每周3次降到1次
      - 工作时间反而增加了20%
      可能是过度工作导致的恶性循环。
      建议今晚早睡，明天安排一次运动。"
```

**场景 4: 月度复盘**
```
系统: [每月1号]
     "月度回顾:
      - 健康: 跑步12次共60km，体重下降1.2kg
      - 学习: 读完2本书，课程进度80%
      - 工作: 完成8个需求，平均每天工作9.5h
      - 情绪: 整体平稳，有3次低谷（都在周三）
      - 洞察: 你的低谷都出现在周三，可能与周会压力有关
      - 建议: 下月尝试调整周三的工作安排"
```

### 2.4 未来可能性

虽然当前是 Self-Use，但这个产品有潜力成为：

**1. 开源框架**
- 其他人可以基于这个框架构建自己的生命助手
- 提供可配置的维度、决策引擎、进化机制

**2. 小范围分享**
- 给家人、朋友使用（独立部署，数据隔离）
- 每个人都有自己的私人助手实例

**3. 企业级应用**
- 团队协作场景（项目管理、知识沉淀）
- 组织级的决策支持系统

但这些都是远期可能性，**当前阶段只专注于服务好一个用户：我自己**。

## 3. 产品架构

### 3.1 五层生命框架

BetterMe 所理解的"生命维度"，不是单一的功能分类，而是一个从经历到秩序的五层结构：

#### 1. 自我 (Self)
关注"我是谁，我正在经历什么"
- 爱
- 孤独
- 勇敢
- 意志
- 坚强
- 恐惧
- 意义感
- 自我认同
- 成长

#### 2. 生活 (Life)
关注"我如何过好每一天"
- 健康
- 学习
- 工作
- 规划
- 习惯
- 节律
- 效率
- 决策

#### 3. 世界 (World)
关注"我如何观察外部世界"
- 对事件的观察
- 对人的理解
- 对关系的体会
- 对社会与现实的判断
- 对变化与规律的感悟

#### 4. 认知 (Cognition)
关注"我如何理解这些经历与观察"
- 判断
- 归因
- 反思
- 心智模型
- 信念更新
- 原则形成

#### 5. 秩序 (Order)
关注"我如何形成长期稳定的内在结构"
- 节律
- 原则
- 方法
- 决策框架
- 自律
- 长期方向
- 内在稳定感

这五层构成一条完整链路：

> **经历 → 感知 → 观察 → 认知 → 秩序**

BetterMe 的目标，不只是记录事实，而是从事实中理解状态，从状态中靠近生命，并最终帮助我形成属于自己的秩序。

## 5. 跨维度智能 (Phase 3)

### 5.1 关联分析

系统能够发现不同维度之间的关联:

| 关联类型 | 示例 |
|---------|------|
| 健康 ↔ 工作 | "你运动后的工作效率平均提升30%" |
| 情绪 ↔ 健康 | "睡眠不足6小时的第二天，焦虑概率上升50%" |
| 学习 ↔ 情绪 | "你在心情好的时候学习效率更高" |
| 工作 ↔ 情绪 | "连续加班3天后情绪明显下降" |

### 5.2 整体平衡建议

```
"本周分析:
 - 工作时间偏多 (52h)，运动偏少 (1次)
 - 睡眠质量下降，情绪波动增大
 - 建议: 明天安排一次运动，本周末留出完整休息时间"
```

### 5.3 人生复盘

```
"月度回顾:
 - 健康: 跑步4次共20km，体重下降0.5kg，进度正常
 - 学习: 读完1本书，课程进度60%，略慢于计划
 - 工作: 完成3个需求，1个延期，效率中等
 - 情绪: 整体平稳，周三有一次低谷（加班导致）
 - 建议: 下月重点提升学习时间分配"
```

## 6. 交互设计

### 6.1 交互原则

1. **自然语言优先**: 用户用最自然的方式表达，系统负责理解
2. **最少操作**: 一句话完成记录，无需打开多个页面
3. **确认而非猜测**: 关键操作需用户确认，避免误操作
4. **主动但不打扰**: 智能提醒在合适的时机出现

### 6.2 对话风格

- 温暖但不油腻
- 专业但不说教
- 简洁但不冷漠
- 有个性但不抢戏

### 6.3 核心交互流程

**记录流程**:
```
用户: "今天跑了5公里"
系统: [识别意图: 运动记录]
      [槽位检测: 类型=跑步, 距离=5km, 缺少时长]
系统: "跑了多久呀？"
用户: "30分钟"
系统: [补全槽位: 时长=30min]
系统: "记录一下: 跑步 5.0km 30分钟，对吗？"
用户: "对"
系统: "已记录！配速6分钟/km，比上次快了10秒 💪"
```

**主动建议流程**:
```
系统: [每日提醒]
      "早上好！昨天睡了6小时，比平时少1.5小时。
       今天建议轻量训练或休息，别硬撑。"
```

**跨维度洞察流程**:
```
用户: "最近感觉效率很低"
系统: [分析近2周数据]
      "看了一下你最近的数据:
       - 连续5天睡眠不足7小时
       - 运动频率从每周3次降到1次
       - 工作时间反而增加了20%
       可能是过度工作导致的恶性循环。
       建议今晚早睡，明天安排一次运动。"
```

### 7.2 维度扩展的技术策略

每个新维度的接入遵循统一模式:

1. **Schema 定义**: 在 `schemas/` 新增意图枚举和 Payload 模型
2. **NLU 扩展**: 在 `prompts/nlu.py` 新增意图规则和 few-shot
3. **Preprocess 扩展**: 在 `nlu/preprocess.py` 新增领域关键词提取
4. **Record 实现**: 在 `record/` 新增落库函数
5. **Decision 实现**: 在 `decision/` 新增领域决策逻辑
6. **Display 扩展**: 在 `chat/display.py` 新增展示格式

## 8. 版本规划

### Phase 1: 健康维度深化 (v2.1, 4周)

**目标**: 把健康维度从"能记录"升级到"能决策"

**P0 需求**:
- [ ] FR-01: 运动建议引擎 — 基于历史运动数据和目标，生成每日训练建议
- [ ] FR-02: 主动提醒 — 久未记录提醒、目标偏离提醒、恢复提醒
- [ ] FR-03: 趋势分析 — 体重趋势、运动量趋势、配速趋势可视化描述

**P1 需求**:
- [ ] FR-04: 饮食建议 — 基于目标和当天摄入给出饮食建议
- [ ] FR-05: 周报生成 — 每周自动生成健康维度周报
- [ ] FR-06: AutoAgent 事件记录上线 — 打通 Action Event 写入

**验收标准**:
- 运动建议准确率 > 70%（用户采纳率）
- 主动提醒触达率 > 90%
- 周报生成覆盖率 100%

### Phase 2: 多维度扩展 (v2.2, 6周)

**目标**: 扩展学习、工作、情绪三个维度的记录和基础决策

**P0 需求**:
- [ ] FR-07: 学习维度 — 读书/课程/技能记录 + 复习提醒
- [ ] FR-08: 工作维度 — 任务/会议/复盘记录 + 时间分析
- [ ] FR-09: 情绪维度 — 情绪/能量记录 + 模式识别

**P1 需求**:
- [ ] FR-10: AutoAgent EMO 上线 — 弹性记忆组装器投产
- [ ] FR-11: 多维度查询 — "今天做了什么"跨维度汇总
- [ ] FR-12: 维度间基础关联 — 运动与情绪、睡眠与效率

**验收标准**:
- 新维度意图识别准确率 > 85%
- 用户日均记录维度数 >= 2
- EMO 上线后平均 token 下降 >= 20%

### Phase 3: 跨维度智能 (v2.3, 6周)

**目标**: 实现维度间关联分析和整体生命优化

**P0 需求**:
- [ ] FR-13: 跨维度关联分析引擎
- [ ] FR-14: 整体平衡建议
- [ ] FR-15: 月度生命复盘

**P1 需求**:
- [ ] FR-16: AutoAgent Evolution 灰度上线
- [ ] FR-17: 目标体系管理（短/中/长期目标联动）
- [ ] FR-18: 习惯追踪与分析

**验收标准**:
- 关联分析洞察被用户认可率 > 60%
- 月度复盘生成覆盖率 100%
- Evolution patch 回滚率 < 10%

### Phase 4: 生命操作系统 (v3.0, 12周+)

**目标**: 成为真正的个人生命助手

- 价值观对齐: 理解用户核心价值观，所有建议基于此
- 人生路径规划: 基于历史数据的长期规划建议
- 更多维度: 财务、人际、创作
- 多模态输入: 图片（食物识别）、语音

## 9. 指标体系

### 9.1 北极星指标

**我的持续使用天数** — 衡量产品是否真正融入我的生活

### 9.2 核心指标

| 指标 | 定义 | 目标 |
|------|------|------|
| 日均记录数 | 每天记录的条数 | >= 3 |
| 建议采纳率 | 主动建议被我采纳的比例 | > 50% |
| 任务一次完成率 | 意图一次理解正确的比例 | > 85% |
| 数据完整度 | 各维度数据记录的完整性 | > 80% |

### 9.3 过程指标

| 指标 | 定义 | 目标 |
|------|------|------|
| 意图识别准确率 | NLU 正确识别意图的比例 | > 90% |
| 平均对话步骤数 | 完成一次记录的平均轮次 | < 3 |
| 平均响应时间 | 系统回复延迟 | < 2s |
| 二次追问率 | 需要追问补充信息的比例 | < 30% |

### 9.4 健康指标

| 指标 | 定义 | 目标 |
|------|------|------|
| 误操作率 | 错误记录/删除的比例 | < 2% |
| patch 回滚率 | 认知更新回滚的比例 | < 10% |
| 系统可用性 | 服务正常运行时间 | > 99.5% |

**注**: 所有指标都是为了衡量"这个系统对我自己是否有用"，而不是为了商业化或规模化。

## 10. 风险与应对

### 10.1 产品风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 我自己记录疲劳 | 使用中断，数据断层 | 降低记录门槛，允许模糊表达和被动采集 |
| 建议不准确 | 我对系统失去信任 | 先观察后建议，保守策略优先，保留人工判断权 |
| 维度过多导致复杂 | 系统变重，难以长期维护 | 先聚焦健康，再逐步扩展 |
| 隐私与数据安全 | 不敢记录真实信息 | 本地优先，数据加密，完全自我掌控 |
| 系统维护成本过高 | 产品半途而废 | 控制复杂度，优先做高频高价值能力 |

### 10.2 技术风险

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM 成本过高 | 运营成本不可控 | 确定性规则优先，LLM 只做最后一公里 |
| 认知进化失控 | 系统行为退化 | shadow → applied 双阶段 + 可回滚 |
| 上下文膨胀 | 响应变慢 | EMO 弹性记忆控制 token 预算 |
